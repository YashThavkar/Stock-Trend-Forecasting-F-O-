# Stock Trend Forecasting (F&O)

Models **implied-volatility-style surface** dynamics from `training_data.csv`: each row is a trading **Date** and option **Tenor**, with **moneyness** columns `0.1`–`1.9`. The Jupyter notebook (`final_prediction.ipynb`) builds a panel, adds lag and calendar features, fits **HistGradientBoostingRegressor**, validates on recent dates, and fills `prediction_template.csv` → `predictions_filled.csv`.

Educational / portfolio use only — not investment advice.

## Portfolio website (GitHub Pages)

This repo includes a static demo under **`docs/`**: interactive Plotly charts (when `data.json` loads), PNG fallbacks, surface lookup demo, EDA HTML, and profiling report. Anyone with the link can use it in a browser — no Python required.

### Go live (pick one method)

**Option A — Deploy from branch (simplest)**

1. Push this repository to GitHub (include the whole `docs/` tree: `data.json`, `demo-data.js`, images, HTML).
2. On GitHub: **Settings → Pages → Build and deployment**.
3. **Source**: **Deploy from a branch**.
4. Branch: **`main`** (or your default), folder: **`/docs`**, save.

**Option B — GitHub Actions (auto-publish on every push to `main`)**

1. Push the repo, including `.github/workflows/deploy-pages.yml`.
2. **Settings → Pages**: set **Source** to **GitHub Actions** (not “Deploy from a branch”).
3. The **Deploy GitHub Pages** workflow uploads the `docs/` folder. Open the workflow run to see the **Page URL** when it succeeds.

A **`docs/.nojekyll`** file is included so GitHub Pages does not run Jekyll on your static files.

### Local preview

Browsers block `fetch()` on `file://` for chart data, so serve `docs/` locally:

```bash
cd docs
python -m http.server 8080
```

Open `http://localhost:8080`. The surface lookup still works from `file://` thanks to embedded `demo-data.js`.

### Refresh chart data

After updating CSVs or regenerating predictions:

```bash
python scripts/export_portfolio_data.py
```

Commit the updated `docs/assets/data.json`.

### Static images and training data HTML

PNG charts and `docs/training_data.html` (tables + summary from the CSV) are generated by:

```bash
python scripts/build_docs_assets.py
```

Commit new files under `docs/assets/images/` and `docs/training_data.html` when you change the data.

The same command writes:

- **`docs/assets/demo_lookup.json`** — used when the site is served over `http/https`
- **`docs/assets/demo-data.js`** — embeds the same data so the **Surface lookup** works when you open `docs/index.html` via **`file://`** (no local server needed for that widget)
- **`docs/training_profile.html`** — full **Pandas Profiling–style** report (histograms, correlations, alerts) via **ydata-profiling** if installed

Install profiling dependency once:

```bash
pip install ydata-profiling
```

If `ydata-profiling` is missing, the build still succeeds and writes a **stub** `training_profile.html` with install instructions.

**Which HTML is which**

| File | What it is |
|------|------------|
| `docs/training_data.html` | Light tables + `describe()` only |
| `docs/training_profile.html` | Real ydata-profiling UI (like your Pandas Profiling screenshots) |

## Requirements

Install everything (notebook, asset scripts, and the optional web server for hosting):

```bash

```

Run `final_prediction.ipynb` from the project root.

Optional profiling (not in `requirements.txt` by default — large dependency):

```bash
pip install ydata-profiling
```

## Host the demo on Render

The repo includes **`main.py`**, which serves the static files under **`docs/`** with Flask + Gunicorn.

1. Create a **Web Service** on [Render](https://render.com), connect this repository.
2. **Runtime**: Python 3.
3. **Build command**: `pip install -r requirements.txt`
4. **Start command**: `gunicorn main:app --bind 0.0.0.0:$PORT`  
   (A **`Procfile`** is included with the same command if Render picks it up automatically.)

Open the service URL; you should see the same site as GitHub Pages. Ensure `docs/` is committed (including `assets/data.json`, images, and HTML).
